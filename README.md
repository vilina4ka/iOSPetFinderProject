# Лапки — приложение для поиска потерявшихся питомцев

iOS-приложение на Swift + бэкенд на Go. Помогает владельцам потерявшихся животных находить их быстрее: публикация объявлений на карте, репорты о наблюдениях, чат между владельцем и очевидцем, уведомления и подписки на чужих питомцев.

## Стек технологий

**iOS (клиент)**
- Swift 5, UIKit
- Архитектурный паттерн MVVM (+ Repository)
- Yandex MapKit Mobile SDK — карта, геометки, маршруты
- Kingfisher — асинхронная загрузка и кэширование изображений
- KeychainAccess — безопасное хранение JWT-токена
- Combine — реактивная подписка на изменения состояния авторизации

**Backend**
- Go, Gin
- PostgreSQL 16 + расширение PostGIS 3.4
- pgx/v5 — нативный драйвер PostgreSQL для Go
- JWT (HS256) — авторизация запросов
- gorilla/websocket — реалтайм-чат через WebSocket
- Yandex OAuth 2.0 — аутентификация пользователей
- aws-sdk-go-v2 — загрузка фото в Yandex Object Storage (S3-совместимый API)
- swaggo/swag + gin-swagger — автогенерация Swagger-документации

**Инфраструктура**
- Сервер: VPS в Yandex Cloud (Ubuntu), порт 8080
- Хранение фотографий: Yandex Object Storage (S3-совместимый)
- Контейнеризация: Docker + Docker Compose

## Архитектура бэкенда

Построен по трёхуровневой схеме **Handler → Service → Repository**.

- **Handler** — принимает HTTP-запрос, валидирует входные данные, вызывает сервис, возвращает JSON. Бизнес-логики не содержит. Исключение: `AuthHandler` работает напрямую с пулом соединений (`pgxpool.Pool`), так как логика аутентификации тесно завязана на Yandex OAuth и не вынесена в отдельный репозиторий.
- **Service** — бизнес-логика: проверка прав доступа, оркестрация нескольких репозиториев, асинхронная отправка уведомлений через горутины (`go func()`).
- **Repository** — только SQL-запросы. Каждый репозиторий реализует Go-интерфейс, что позволяет подменять реализацию при тестировании.

```
GET /pets?lat=55.7&long=37.6&radius=5000
         │
    PetsHandler.ListPets()
         │
    PetService.ListPets()          ← фильтрация по типу/породе/дате
         │
    PostgresPetRepo.ListInRadius() ← SQL: ST_DWithin (PostGIS)
```

## Архитектура iOS-приложения

Паттерн **MVVM + Repository**, четыре слоя:

- **Model** — Swift-структуры (`Codable`). Декодер настроен с `.convertFromSnakeCase`, поэтому `owner_id` → `ownerId` происходит автоматически.
- **ViewModel** — `@MainActor final class`, реализует `ObservableObject`. Хранит `@Published`-состояние, содержит бизнес-логику и вызовы через `async/await`. Не знает об API напрямую — работает только через протокол репозитория.
- **Repository** — протокол + реализация над `APIClient`. Изолирует ViewModel от деталей сетевого слоя; при необходимости можно подменить на мок-реализацию.
- **View (ViewController)** — отвечает только за отрисовку и обработку жестов. Подписывается на `@Published`-свойства ViewModel через Combine (`.sink`), что позволяет UI реагировать на изменения автоматически. Для разовых действий (загрузка, отправка) запускает `Task { }`.


## Ключевые процессы

### 1. Авторизация через Яндекс

1. Приложение открывает страницу Яндекс OAuth в Safari через `ASWebAuthenticationSession`.
2. После входа Яндекс делает редирект на `lapki://auth?code=...`.
3. `SceneDelegate` перехватывает deep link, извлекает `code`.
4. Приложение отправляет код на бэкенд: `POST /auth/yandex`.
5. Бэкенд обменивает код на access token у Яндекса, запрашивает профиль (`login.yandex.ru/info`), делает `UPSERT` в таблицу `users`, генерирует JWT (срок жизни 30 дней) и возвращает его.
6. JWT сохраняется в Keychain. Все последующие запросы отправляют заголовок `Authorization: Bearer <token>`.
7. `AuthManager` — синглтон с `@Published var currentUser`. `SceneDelegate` подписывается через Combine и переключает root VC между экраном входа и главным TabBar.

### 2. Геопоиск питомцев

1. `LocationService` запрашивает разрешение, получает координаты устройства.
2. `PetListViewModel` вызывает `GET /pets?lat=&long=&radius=5000` (плюс опциональные фильтры).
3. На бэкенде SQL-функция `pets_in_radius` использует PostGIS-оператор `ST_DWithin` — работает с географическими объектами и учитывает кривизну Земли. Поле `pets.location` типа `GEOGRAPHY(POINT, 4326)` обновляется триггером при каждом `INSERT`/`UPDATE`.
4. Дополнительная фильтрация по типу, породе и периоду пропажи происходит в сервисном слое.
5. Клиент отображает результаты в `UITableView` (список) и на карте Яндекса (пины).

### 3. Репорты о наблюдениях

Репорт — заявка от пользователя «я видел этого питомца вот здесь».

1. Очевидец открывает карточку питомца → нажимает «Я видел» → `ReportSightingViewController`.
2. Указывает точку на карте через `LocationPickerViewController`, добавляет комментарий и опционально фото.
3. `POST /pets/:id/sightings` → `PetService.AddSighting()`:
   - Репорт сохраняется в таблицу `sightings` со статусом `pending`.
   - Владельцу асинхронно через горутину создаётся уведомление типа `sighting`.
4. Владелец видит репорт в разделе «Наблюдения» и меняет статус на `confirmed` или `rejected`.
5. При подтверждении подписчики питомца получают уведомление о новом месте.
6. На карточке подтверждённые наблюдения отображаются пинами, между точкой пропажи и точкой наблюдения рисуется пунктирная линия `YMKPolyline`.

### 4. Чат (WebSocket + REST)

Чат привязан к паре «питомец + два пользователя» (владелец и очевидец).

- **История**: `GET /chats/:petID?other_user_id=&limit=&offset=` — пагинированный список сообщений.
- **Отправка**: `POST /chats/:petID/message` — сервер сохраняет сообщение в БД и асинхронно пытается доставить его через Hub.
- **Реалтайм**: `GET /ws` — единое WebSocket-соединение на пользователя. Все входящие сообщения приходят без polling. Hub хранит `map[userID]*Client` с `sync.RWMutex`; доставка — неблокирующий `select/default`.
- **Редактирование / удаление**: `PATCH /chats/message/:messageID`, `DELETE /chats/message/:messageID`.
- **Прочитано**: `POST /chats/:petID/read` — отметить все сообщения треда прочитанными.
- **Файлы и фото**: сначала загружаются через `POST /upload` (Yandex Object Storage), затем URL передаётся в тело сообщения.
- **Список чатов**: `GET /chats` — SQL с оконными функциями выбирает последнее сообщение каждого треда + JOIN с `users` для аватара собеседника.

### 5. Уведомления

Уведомления хранятся в таблице `notifications`. Push-уведомления (APNs) не реализованы — центр уведомлений работает как inbox: история событий, доступная при открытии вкладки.

Типы уведомлений:

| Тип | Когда создаётся |
|---|---|
| `sighting` | Кто-то сообщил о наблюдении / владелец подтвердил репорт |
| `message` | Новое сообщение в чате |
| `update` | Владелец обновил карточку питомца (приходит подписчикам) |

При тапе `NotificationsViewController` навигирует на конкретный репорт, чат или карточку питомца — в зависимости от `type` и `sighting_id`.

### 6. Подписки

Пользователь подписывается на любого чужого питомца (`POST /pets/:id/subscribe`). Подписчики получают уведомления при обновлении карточки и при подтверждении нового наблюдения.

### 7. Загрузка фотографий

`POST /upload` принимает `multipart/form-data`, кладёт файл в Yandex Object Storage и возвращает публичный URL. На клиенте загрузку инкапсулирует `StorageManager` — синглтон, который вызывает `APIClient.upload` и возвращает готовый URL.

### 8. Генерация постера

Постер генерируется полностью на клиенте через `UIGraphicsImageRenderer` — без обращения к серверу. Поддерживаются четыре формата: квадрат для соцсетей, 9:16 для stories, A4 и A3 для печати. QR-код создаётся через `CIFilter(CIQRCodeGenerator)` без сторонних библиотек и ведёт на веб-страницу питомца (`GET /view/pets/:id`), которую отдаёт бэкенд.

## База данных

| Таблица | Назначение |
|---|---|
| `users` | Профили пользователей (Yandex OAuth) |
| `pets` | Объявления о потерявшихся питомцах |
| `sightings` | Репорты о наблюдениях |
| `messages` | Сообщения в чатах |
| `notifications` | Центр уведомлений |
| `subscriptions` | Подписки пользователей на питомцев |

`pets.location` имеет тип `GEOGRAPHY(POINT, 4326)` и обновляется триггером `pet_location_trigger`. Геопоиск реализован через SQL-функцию `pets_in_radius` с PostGIS.

Миграции применяются автоматически при старте контейнера через `scripts/entrypoint.sh`.

## API — основные эндпоинты

Все эндпоинты кроме `/auth/*` и `/view/*` требуют заголовок `Authorization: Bearer <JWT>`.

Swagger UI доступен по адресу: `http://178.154.198.101:8080/swagger/index.html`

## Запуск бэкенда
Бэкенд задеплоен на сервер, но можно достать его локально.

### Docker

```bash
# Создать .env на основе примера
cp backend/.env.example backend/.env
# Заполнить значения в .env

cd backend
make up        # поднять БД + бэкенд
make logs      # смотреть логи
make restart   # пересобрать и перезапустить после изменений
make down      # остановить
```

Миграции применяются **автоматически** при старте контейнера.
